# Survey — 2026-07-03 — scope: DEEP

Pre-structural-change review of the staged (uncommitted) Sprint 1: AUDIT_ENFORCE=1
flip, only-Gaffer-commits guard, deterministic Skeptic sampler, trust-badge backfill.

## Executive read
The provenance-stamp mechanics are sound and I verified them by measurement: the
commit-authorization marker propagates through `exec flock … git` to the hook, the
trust badge is hash-neutral (identical content hash badged vs unbadged), badge counts
are honest (spot-checked 4 docs, all match), and the weekly harness only commits the
HOF pages it records verdicts for. One change is load-bearing and worth fixing before
commit: the new deterministic Skeptic sampler is **blind to the `**[data: …]**`
spec-form tag** — on the dustin-martin doc it probes zero of 148 tags while the badge
advertises "All 148 verified." It doesn't let wrong numbers ship (DataSentinel still
checks every tag), but the redundant Skeptic gate silently self-disables on exactly
the doc class I1 is steering toward. Everything else is ship-quality.

## Ranked findings

### F1 — Skeptic deterministic sampler is blind to spec-form `**[data: …]**` tags  [HIGH] [class: 4 / NEW]
- Evidence: `scripts/skeptic_sample_tags.py` regex is `\*\*\[(data|historical record)\]\*\*`
  (plain bold only). `python3 scripts/skeptic_sample_tags.py docs/news/2026-06-21-dustin-martin-the-storm.md`
  → "# no verifiable [data] tags" (exit 0). That doc has 0 plain + 148 spec-form tags.
  Meanwhile `inject_trust_badge.py` deliberately counts all three forms → badges "All 148".
  The two scripts disagree on what a `[data]` tag is.
- Impact: Skeptic step-7 smoke test becomes a silent no-op on any spec-form doc — a
  redundant gate reporting PASS with zero coverage, on the tag form the I1 spike promotes.
- Owner: Gaffer (gate coverage; Scientist executes the regex reconciliation).
- Recommended outcome: sampler recognizes the same three forms `inject_trust_badge`
  counts; a doc with tags-but-zero-plain must not return empty.
- Effort: S · rank 1

### F2 — HOF stamp claims "DataSentinel: PASS"; backing record is authored by check_hof_numbers  [MEDIUM] [class: 4]
- Evidence: `check-council-stamp.sh:verify_stamp_against_audit` checks only doc_path,
  doc_hash, verdict==PASS — never agent_id. `weekly_refresh.sh` Phase 2b writes
  `record-sentinel-verdict.sh … --agent check_hof_numbers`. Gate certifies "some council
  process PASSed this exact content," not "DataSentinel verified it."
- Impact: provenance attribution is decorative; the stamp text overclaims DataSentinel
  on deterministically-generated pages. The record is honest (not simulated) — the
  inaccuracy lives in the stamp.
- Owner: Gaffer.
- Recommended outcome: ship as-is is acceptable (content-integrity guarantee is real,
  check_hof_numbers is a genuine deterministic gate); either name the real verifier in
  the HOF stamp or have the gate bind agent identity. Not a blocker.
- Effort: S · rank 3

### F3 — Enforcement depends on gitignored, local-only records; non-primary environments brick  [MEDIUM] [class: NEW]
- Evidence: `.gitignore` adds `.claude/audit/sentinel-*.json`; `AUDIT_ENFORCE` default→1.
  On a fresh clone / CI / second machine with no records, committing ANY stamped doc
  hard-fails ("no audit record exists for it"). Compounds memory item project_githooks_gap.
- Impact: the primary machine self-heals (weekly refresh regenerates HOF records each
  run; news committed interactively with live DataSentinel records), but any other env
  is blocked from committing council docs except via `--no-verify` / `AUDIT_ENFORCE=0`.
  Also: recommitting a published news doc without re-running DataSentinel now hard-fails
  (intended, but a behavioral change to flag).
- Owner: Gaffer.
- Recommended outcome: document the AUDIT_ENFORCE=0 bootstrap for record-less environments
  and make the fresh-clone behavior explicit; consider a record-regeneration lane.
- Effort: S · rank 4

### F4 — only-Gaffer marker authorizes but does not serialize  [LOW] [class: 8]
- Evidence: pre-commit guard checks only `COUNCIL_COMMIT_AUTHORIZED`; the `flock` lives
  only in `git_commit_safe.sh`. A process that `export COUNCIL_COMMIT_AUTHORIZED=1 &&
  git commit` satisfies the guard while skipping the lock.
- Impact: blocks accidental unmarked commits (good) but is a speed-bump, not a hard
  single-writer serializer.
- Owner: Gaffer. Effort: S · rank 5

## Anti-pattern list (additions)
- A deterministic gate must probe the same surface the LLM instruction did — verify
  coverage on every tag FORM, not just the common one. (F1)
- Enforcement keyed on gitignored local state is non-portable; the gate's behavior
  silently changes by environment. (F3)
- An authorization marker is not a lock — serialize with flock; don't rely on a boolean
  env var for single-writer. (F4)
